Compare Methods, Fix Common Pitfalls

You've deployed OpenClaw — now understand why things break and how to fix them. This workshop compares four deployment methods side by side, then dives into the most common pitfalls: gateway crashes, token mismatches, region mapping errors, and port exposure issues. You'll leave with a troubleshooting playbook and the confidence to debug any deployment.

Who It's For
Developers who have deployed OpenClaw at least once and want to understand tradeoffs and fix issues faster
What You Get
A mental model for choosing deployment methods plus a troubleshooting playbook for common failures
Time Required
~45 minutes covering comparison, debugging exercises, and health check patterns

What You'll Build

  • A comparison framework for evaluating local vs cloud, CPU vs GPU serverless deployments
  • A troubleshooting playbook covering the top failure modes in OpenClaw deployments
  • Health check scripts you can reuse across any deployment method

What We'll Cover

  • Local vs Cloud deployment — when to run on your machine vs Nebius compute, and the tradeoffs of each
  • CPU vs GPU serverless — cost, latency, and model availability differences
  • Gateway crash diagnosis — reading logs, identifying OOM kills, restart loops, and WebSocket failures
  • Token mismatch debugging — API key formats, environment variable precedence, and auth flow tracing
  • Region mapping errors — Token Factory region availability, endpoint URLs, and latency implications
  • Port exposure and firewall issues — cloud security groups, local firewalls, bind address mistakes
  • Health check patterns — automated scripts to verify gateway, Token Factory, and end-to-end connectivity

Step-by-Step Guide

1
Map the Four Deployment Methods 10 min

Understand the four deployment paths and when each one makes sense.

  1. Review the four methods: Local Install (npm), Web Console (browser), Install Scripts (CLI), Docker (container)
  2. Compare on five axes: setup time, repeatability, isolation, resource requirements, and production readiness
  3. Identify which method fits your current use case (prototyping, demo, staging, production)
# Quick comparison matrix # Method Setup Repeatable Isolation Prod-Ready # Local npm ~5 min Low None No # Web Console ~20 min Medium VM Yes # Install Script ~15 min High VM Yes # Docker ~10 min High Container Yes
Tip: Local npm is best for learning and prototyping. Install scripts are best for repeatable cloud deployments. Docker adds isolation but requires Docker knowledge.
Checkpoint: You can articulate why you'd choose one method over another for a given scenario.
2
Debug Gateway Crashes & Token Mismatches 15 min

Walk through the most common failure modes and learn how to diagnose each one.

  1. Check gateway logs for crash indicators: OOM kills, unhandled promise rejections, and bind errors
  2. Trace a token mismatch: verify TOKEN_FACTORY_API_KEY format, check env var precedence, test with curl
  3. Simulate a region mapping error by pointing to the wrong Token Factory endpoint
  4. Fix each issue and verify with a health check
# Check for gateway crashes openclaw logs --tail 50 | grep -i "error\|crash\|killed"
# Test Token Factory connectivity directly curl -s -H "Authorization: Bearer $TOKEN_FACTORY_API_KEY" \ https://api.tokenfactory.nebius.com/v1/models | jq '.data[0].id'
# Check environment variable precedence env | grep -E "TOKEN_FACTORY|OPENCLAW|INFERENCE"
Tip: The most common token mismatch is using a Token Factory key that starts with v1. in a field that expects the gateway token (or vice versa). They are different credentials.
Checkpoint: You can identify and fix a gateway crash, token mismatch, and region error from logs alone.
3
Fix Port Exposure & Firewall Issues 10 min

Diagnose and resolve connectivity problems caused by port binding, firewalls, and security groups.

  1. Verify the gateway bind address: loopback (local only) vs 0.0.0.0 (all interfaces)
  2. Check local firewall rules (iptables/ufw on Linux, pf on macOS)
  3. Verify cloud security group rules in the Nebius console
  4. Test external connectivity with curl from another machine
# Check what the gateway is bound to ss -tlnp | grep 18789
# Test external connectivity curl -s http://{your-instance-ip}:18789/health
# If bound to loopback, rebind to all interfaces for cloud access openclaw gateway --bind 0.0.0.0 --auth token --token $OPENCLAW_GATEWAY_TOKEN
Tip: On Nebius, you need to open port 18789 in both the VM's local firewall AND the cloud security group. Missing either one will block external access.
Checkpoint: You can access the Control UI from an external browser after resolving firewall and bind address issues.
4
Build a Health Check Script 10 min

Create a reusable script that validates gateway, Token Factory, and end-to-end connectivity.

  1. Write a script that checks gateway process status
  2. Add a Token Factory API connectivity test
  3. Add an end-to-end message test (send a prompt, verify a response)
  4. Output a clear pass/fail summary
#!/bin/bash # health-check.sh — OpenClaw deployment health check echo "=== Gateway Status ===" curl -sf http://localhost:18789/health && echo " OK" || echo " FAIL" echo "=== Token Factory ===" curl -sf -H "Authorization: Bearer $TOKEN_FACTORY_API_KEY" \ https://api.tokenfactory.nebius.com/v1/models > /dev/null \ && echo "OK" || echo "FAIL" echo "=== End-to-End ===" openclaw status && echo "OK" || echo "FAIL"
Tip: Run this script after every deployment or configuration change. Consider adding it to a cron job for continuous monitoring.
Checkpoint: Your health check script reports pass/fail for all three checks.
