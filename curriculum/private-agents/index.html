Step-by-Step Guide

1
Install the Nebius CLI 5 min

Install and configure the Nebius AI Cloud CLI for managing serverless endpoints and dedicated infrastructure.

  1. Download and install the Nebius CLI using the install script
  2. Authenticate with your Nebius AI Cloud account
  3. Verify your configuration is correct
# Install Nebius CLI curl -sSL https://storage.eu-north1.nebius.cloud/cli/install.sh | bash
# Login to your Nebius account nebius auth login
# Verify your config cat ~/.nebius/config.yaml
Checkpoint: Running nebius iam whoami returns your user info without errors.
2
Set Up Private Networking 5 min

Configure a VPC network and subnet for your private agent deployment. This keeps all traffic off the public internet.

  1. List your existing VPC networks to confirm access
  2. Get the subnet ID for your default subnet
  3. Export it as an environment variable for later steps
# List available networks nebius vpc network list
# Get your subnet ID and export it export SUBNET_ID=$(nebius vpc subnet get-by-name \ --name default-subnet --format jsonpath='{.metadata.id}') echo $SUBNET_ID
Tip: Private networking ensures your agent traffic never traverses the public internet. This is a requirement for most compliance frameworks handling sensitive data.
Checkpoint: You have SUBNET_ID exported and ready for deployment.
3
Deploy OpenClaw on Serverless 10 min

Deploy the OpenClaw agent on a Nebius Serverless endpoint. This runs on CPU-only infrastructure — inference is handled by your dedicated Token Factory endpoint in the next step.

  1. Generate a secure authentication token for your endpoint
  2. Deploy OpenClaw using the Nebius serverless CLI
  3. Retrieve and export the endpoint ID for later use
# Generate a secure auth token export AUTH_TOKEN=$(openssl rand -hex 32)
# Deploy OpenClaw on Nebius Serverless nebius msp serverless v1alpha1 endpoint create \ --name openclaw-private \ --container-image openclaw:latest \ --container-template-resources-platform cpu-d3 \ --container-template-resources-preset 4vcpu-16gb \ --port 8080 \ --username admin \ --password "$AUTH_TOKEN" \ --network-id <your-network-id> \ --parent-id <your-project-id>
# Get your endpoint ID export ENDPOINT_ID=$(nebius msp serverless v1alpha1 endpoint get-by-name \ --name openclaw-private --format jsonpath='{.metadata.id}')
Tip: For production, consider disabling --public and using private networking only. Replace <your-network-id> and <your-project-id> with your actual Nebius resource IDs.
Checkpoint: Your OpenClaw endpoint is active and responding to health checks.
4
Create a Dedicated Token Factory Endpoint 10 min

Set up a dedicated Token Factory endpoint with zero data retention. Isolated inference — your prompts never touch shared infrastructure.

  1. Set your dedicated Token Factory API key and endpoint URL
  2. Test the endpoint with a simple completion request
  3. Verify zero-retention is enabled in the Nebius console
# Set your dedicated TF endpoint credentials export TF_API_KEY=<your-dedicated-tf-api-key> export TF_ENDPOINT=https://<your-dedicated-endpoint>.nebius.com
# Test the zero-retention endpoint curl $TF_ENDPOINT/v1/chat/completions \ -H "Authorization: Bearer $TF_API_KEY" \ -H "Content-Type: application/json" \ -d '{ "model": "meta-llama/Meta-Llama-3.1-70B-Instruct", "messages": [ { "role": "user", "content": "Test message - verify zero retention" } ], "max_tokens": 50 }'
Tip: Dedicated endpoints have stable latency because you are not sharing capacity with other users. Zero retention means your prompts and responses are never written to disk — they exist only in memory during processing.
Checkpoint: Your dedicated endpoint returns completions and you have confirmed zero-retention is enabled in the console.
5
Build a Private Document Pipeline 15 min

Create an agent workflow that processes sensitive documents with full privacy guarantees. No data persists after processing.

  1. Choose a document type: legal contracts, medical records, or financial reports
  2. Configure the agent to extract key information from documents
  3. Process sample documents through the pipeline
  4. Verify that no data persists after processing
# Example: process a legal contract through the pipeline curl -X POST http://localhost:8080/v1/process \ -H "Authorization: Bearer $AUTH_TOKEN" \ -H "Content-Type: application/json" \ -d '{ "document_type": "legal_contract", "file": "sample-nda.pdf", "extraction": [ "parties", "effective_date", "termination_clause", "obligations" ] }'
# Verify no data persists on the endpoint curl http://localhost:8080/v1/storage/check \ -H "Authorization: Bearer $AUTH_TOKEN" # Should return: {"stored_documents": 0, "cached_prompts": 0}
Tip: Use the sample documents we provide, or bring your own (redacted) examples. The extraction pipeline is the same regardless of document type — the prompts change, not the architecture.
Checkpoint: Your agent processes 3 sample documents and returns accurate extractions. No data remains on the endpoint after processing.
6
Add Audit Logging 10 min

Set up audit logging so your security team can verify what happened without seeing the actual data. Good audit logs answer "who did what when" without revealing "what was in the document."

  1. Enable structured logging on your OpenClaw endpoint
  2. Configure log entries to include request IDs, timestamps, and operations — but not document contents
  3. View logs through the Nebius CLI
  4. Export a sample audit log for your compliance team
# View endpoint logs nebius msp serverless v1alpha1 endpoint logs $ENDPOINT_ID
# Check endpoint status and uptime nebius msp serverless v1alpha1 endpoint get $ENDPOINT_ID
Tip: Good audit logs answer "who did what when" without revealing "what was in the document." Log request IDs, timestamps, document types, and operations — never log document contents or extracted data.
Checkpoint: You can pull audit logs that show processing activity without exposing document contents.
7
Architecture Review & Compliance Prep 10 min

Document your architecture for your security team and plan the production rollout. The key architecture: Serverless (CPU) → Dedicated TF Endpoint (zero retention) → no persistent storage.

  1. Review the architecture: Serverless (CPU) → Dedicated TF Endpoint (zero retention) → no persistent storage
  2. Document the data flow for your compliance team
  3. Calculate costs for dedicated endpoints at your expected volume
  4. Save your deployment scripts for repeatable production deployments
# Save your deployment configuration cat > deploy-private-agent.sh <<'EOF' #!/bin/bash set -euo pipefail # Private Agent Deployment Script # Zero-retention OpenClaw on Nebius Serverless export AUTH_TOKEN=$(openssl rand -hex 32) nebius msp serverless v1alpha1 endpoint create \ --name openclaw-private \ --container-image openclaw:latest \ --container-template-resources-platform cpu-d3 \ --container-template-resources-preset 4vcpu-16gb \ --port 8080 \ --username admin \ --password "$AUTH_TOKEN" \ --network-id $NETWORK_ID \ --parent-id $PROJECT_ID echo "Endpoint deployed. Auth token: $AUTH_TOKEN" EOF chmod +x deploy-private-agent.sh
Tip: The key selling point for compliance: data is never stored at rest, inference happens on dedicated hardware, and you have full audit logs. Most compliance teams approve this architecture quickly because the guarantees are infrastructure-level, not policy-level.
Checkpoint: You have a compliance-ready architecture doc, cost estimates, and a deploy script for production.
