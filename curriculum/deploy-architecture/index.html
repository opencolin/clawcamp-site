Choose the Right Architecture

OpenClaw supports three distinct deployment architectures, each with different cost profiles, latency characteristics, and operational tradeoffs. This workshop walks through all three — Local Gateway + Token Factory, CPU Serverless + Token Factory GPU, and GPU Serverless + Local Model — with text-based architecture diagrams, cost analysis, and clear guidance on when to use each.

Who It's For
Developers and architects deciding how to deploy OpenClaw for production, staging, or cost-optimized workloads
What You Get
A clear understanding of three architecture patterns with cost estimates and decision criteria
Time Required
~30 minutes covering architecture review, cost analysis, and pattern selection

What You'll Build

  • A decision framework for selecting the right deployment pattern based on your requirements
  • Architecture diagrams (text-based) for each of the three patterns
  • A cost model comparing per-token, per-hour, and per-month costs across patterns

What We'll Cover

  • Pattern 1: Local Gateway + Token Factory — gateway runs on your machine or VM, inference via Token Factory API (pay-per-token)
  • Pattern 2: CPU Serverless + Token Factory GPU — gateway runs on serverless CPU compute, inference via Token Factory (lowest ops burden)
  • Pattern 3: GPU Serverless + Local Model — both gateway and model run on GPU serverless (lowest latency, highest cost)
  • Architecture diagrams — text-based flow diagrams showing data paths, network boundaries, and component placement
  • Cost analysis — per-token vs per-hour pricing, idle cost, burst cost, and break-even points
  • When to use each — prototyping, low-traffic production, high-traffic production, latency-sensitive, and cost-sensitive scenarios

Step-by-Step Guide

1
Pattern 1: Local Gateway + Token Factory 8 min

The simplest architecture: run the gateway locally (or on a VM) and use Token Factory for all inference.

  1. Review the architecture: Your Machine/VM runs the OpenClaw Gateway, which calls Token Factory API for inference
  2. Understand the data flow: User → Gateway (local) → Token Factory API → GPU inference → response
  3. Calculate costs: Token Factory charges per-token with no idle cost — you only pay when the agent is responding
  4. Identify best use cases: prototyping, low-traffic bots, development, cost-sensitive workloads
# Pattern 1 Architecture # # [Your Machine / VM] [Nebius Cloud] # +------------------+ +--------------------+ # | OpenClaw Gateway | --API--> | Token Factory | # | (Node.js) | | (GPU Inference) | # +------------------+ +--------------------+ # | Control UI :18789| # +------------------+ # # Cost: ~$0 idle + per-token inference # Latency: Network round-trip to Token Factory (~50-200ms)
Tip: This pattern has zero idle cost. If your agent handles fewer than ~1000 messages/day, per-token pricing is almost always cheaper than reserving GPU compute.
Checkpoint: You can draw this architecture and explain the cost model to someone else.
2
Pattern 2: CPU Serverless + Token Factory GPU 8 min

Run the gateway on serverless CPU compute for zero-ops management, with Token Factory handling GPU inference.

  1. Review the architecture: Gateway runs on Nebius serverless (auto-scales, no VM management), inference via Token Factory
  2. Understand auto-scaling: the gateway scales to zero when idle and spins up on demand
  3. Account for cold starts: first request after idle may take 2–5 seconds while the container starts
  4. Calculate costs: small CPU serverless fee when active + per-token inference — near-zero idle cost
# Pattern 2 Architecture # # [Nebius Serverless (CPU)] [Nebius Cloud] # +------------------+ +--------------------+ # | OpenClaw Gateway | --API--> | Token Factory | # | (Container) | | (GPU Inference) | # +------------------+ +--------------------+ # | Auto-scales 0..N | # | Cold start ~2-5s | # +------------------+ # # Cost: ~$0.01/hr active + per-token inference # Latency: Cold start + Token Factory round-trip
Tip: This is the lowest-ops pattern. No VMs to manage, no SSH, no OS updates. Ideal for production bots that need to be always-available without infrastructure babysitting.
Checkpoint: You understand how cold starts affect user experience and when this tradeoff is acceptable.
3
Pattern 3: GPU Serverless + Local Model 8 min

Run both the gateway and the inference model on GPU serverless for the lowest possible latency.

  1. Review the architecture: Gateway and model run together on GPU serverless — no network hop for inference
  2. Understand model loading: the model loads into GPU memory on cold start (can take 30–60 seconds for large models)
  3. Calculate costs: GPU serverless is priced per-second of GPU time — higher cost but lowest latency
  4. Identify best use cases: latency-sensitive applications, high-throughput workloads, dedicated model requirements
# Pattern 3 Architecture # # [Nebius GPU Serverless] # +-----------------------------------+ # | OpenClaw Gateway + Local Model | # | (Container with GPU access) | # | | # | Gateway --local--> Model (vLLM) | # | No network hop for inference | # +-----------------------------------+ # # Cost: ~$1-4/hr GPU time (even when idle) # Latency: ~10-50ms inference (no network hop)
Tip: This pattern gives the best latency but costs significantly more when idle. Use it for high-traffic production agents or when response time is critical (e.g., real-time voice agents).
Checkpoint: You understand the cost/latency tradeoff and can calculate the break-even point vs Pattern 1.
4
Choose Your Pattern 6 min

Use the decision framework to select the right architecture for your specific use case.

  1. Answer three questions: What is your traffic volume? What is your latency requirement? What is your ops budget?
  2. Use the decision tree: Low traffic + cost-sensitive = Pattern 1. Medium traffic + low ops = Pattern 2. High traffic + low latency = Pattern 3.
  3. Factor in growth: start with Pattern 1 for prototyping, graduate to Pattern 2 for production, use Pattern 3 only when latency demands it
# Decision Tree # # Is this a prototype or dev environment? # YES --> Pattern 1 (Local + Token Factory) # NO --> Do you need sub-100ms inference? # YES --> Pattern 3 (GPU Serverless) # NO --> Do you want zero-ops? # YES --> Pattern 2 (CPU Serverless + TF) # NO --> Pattern 1 on a VM
# Cost Comparison (approximate, 10K messages/day) # # Pattern 1: ~$15-30/month (per-token only) # Pattern 2: ~$20-40/month (CPU serverless + per-token) # Pattern 3: ~$200-800/month (GPU reservation)
Tip: Most teams should start with Pattern 1 and only move to Pattern 2 or 3 when they have a clear reason. Premature optimization of deployment architecture is a common mistake.
Checkpoint: You have selected a pattern for your use case and can justify the choice with cost and latency data.
