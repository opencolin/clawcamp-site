## Phase 2 — Cross-critiques (non-negotiables & objections)

### MVP / Ship-Fast PM
**Must keep:**
- v1.0 NON-NEGOTIABLE (revised, breach forces it): the minimal data-layer lockdown MUST ship before any feature. Concretely: (a) revoke anon SELECT on contacts; (b) change Prefer header in js/supabase.js from return=representation to return=minimal AND have send-verification read the token server-side so verification_token never reaches the client; (c) rewrite the two dashboard PATCH-by-email fetches (dashboard/index.html ~810, ~868) to an auth.uid()/jwt-email-bound update; (d) keep anon INSERT-only on contacts and events so forms keep working. This is still an afternoon-scale change — it does NOT require the full Edge-Function gateway.
- v1.0 NON-NEGOTIABLE (cheap hygiene, unchanged): VERSION file (1.0.0), CHANGELOG.md seeded from git log, capture live schema as supabase/migrations/0001_baseline.sql, scripts/smoke.sh (curl + grep homepage/events/chapters/detail?id=173 for 200 + known string), scripts/rls-probe.sh (assert anon CANNOT read contacts), js/config.js consolidating the duplicated SUPABASE_URL/ANON_KEY, and move the stray PRDs + /safemode + /zusemite + /logo-concepts.html out of the deploy root.
- v1.1 NON-NEGOTIABLE: on-site RSVP. rsvps table (id, event_id, email, name, created_at), INSERT-only RLS mirroring the locked contacts policy, inline email-capture form on events/detail alongside (not replacing) the Luma button, live attendee count via Prefer: count=exact. NO confirmation email, NO reminders, NO calendar buttons in this release — the captured row is the value. This is the single highest-value action on the site and must precede all structured-content and chapter work.
- v1.2 NON-NEGOTIABLE: structured event content. event_speakers / event_schedule / event_sponsors tables FK'd to events, submit-event tabs insert child rows instead of stuffing events.notes, detail page reads tables-first with static EVENT_EXTRAS fallback so id 173 keeps working with ZERO migration. The DB-first-with-static-fallback pattern is the non-negotiable risk-reducer — it means no big-bang cutover.
- v1.3 NON-NEGOTIABLE: chapters become data. chapters table seeded from the two hardcoded cards, /chapters renders from the table, per-chapter view at /chapters/?slug=X via an events.chapter_slug column. Keep the existing search/filter/join-modal JS untouched — swap only the data source.
- v1.4 NON-NEGOTIABLE: minimum gatekeeping. status column (default pending) on events + chapters; public fetches filter to approved (this replaces and DELETES the client-side hardcoded hide-list confirmed at events/index.html ~3411); ONE thin /admin page gated by a small admin-user-ID allowlist in RLS; a honeypot field on public insert forms. Real per-captain RBAC is explicitly OUT of scope until v2.0+.
- v2.0 NON-NEGOTIABLE: retire js/event-extras.js and the static chapter-card fallbacks (only after data paths are proven in prod), one CI workflow running smoke.sh + rls-probe.sh on master push, recap_url column + 'ended' branch on detail, and a one-page operator runbook in /docs so a non-engineer can run submit→approve→add-chapter→read-RSVPs without us.
**Objections:**
- Growth PM v1.0 (sitemap, OG cards, per-event static pre-render, JSON-LD, UTM columns) is the most over-built v1.0 of the five and I object hardest here. A scripts/build-sitemap.js that dumps Supabase + generates one static /events/<slug>/index.html per event introduces a build step this repo does NOT have today (it's static-HTML-on-Vercel, no build). Crawlability for a site with ~10 contacts and a handful of events is premature optimization. Ship a hand-written sitemap.xml + robots.txt + one default og:image if you must — defer the per-event OG-card generator and the build pipeline to v1.5+.
- Platform PM routing ALL public writes through a single submit-form Edge Function in v1.0 is heavier than the breach requires AND the edge function source isn't even in this repo (confirmed: no functions/ dir, send-verification lives Supabase-side only). The minimal fix is: drop anon SELECT on contacts, stop returning verification_token (change Prefer: return=representation to return=minimal in js/supabase.js so the token never crosses the wire), keep anon INSERT-only. That kills the breach in an afternoon WITHOUT building an Edge-Function gateway. Build the gateway later only if spam actually appears.
- Multiple proposals (mine v1.1, Growth v1.3, Community v1.3) say 'reuse the send-verification edge-function pattern' for RSVP confirmation emails. That function is not version-controlled here and confirmation email is NOT load-bearing for capturing an RSVP. Cut all RSVP-email/reminder/digest work from v1.x entirely — the row in the DB is the value; email is v2.0 polish at best.
- Community PM and Platform PM both put a real memberships table + RBAC + captain console as early/mid roadmap. Genuine RBAC (memberships table, SECURITY DEFINER role functions, captain-scoped RLS) is the single most expensive thing any of us proposed. You do NOT need it to capture RSVPs or list chapters. A 5-user-ID admin allowlist in one RLS policy (my v1.4) is enough moderation until there are actually multiple captains to scope. Defer memberships/RBAC to the very end.
- Risk PM v1.1's full quality wall (Playwright suite, html-validate, eslint, Lighthouse CI, Vercel deploy-protection) in the SECOND release is too much ceremony for a static brochure site. The regression guard that actually matters is one thing: scripts/rls-probe.sh asserting anon CANNOT read contacts. Ship that single curl-based check in CI; defer Playwright/Lighthouse/axe to v2.0. The smoke.sh I proposed (6-line curl+grep) covers the rest at 1% of the cost.
- Risk PM v1.4 (Lighthouse Perf>=90 / a11y>=95 / CSP header / self-host third-party / split the 198KB index.html) is a whole release of polish that blocks nothing and protects no data. In ship-fast terms this is a v2.0 'nice to have', not a numbered milestone. The 198KB index.html is not an emergency — it loads fine.
- Growth PM's referral flywheel, leaderboards, K-factor dashboards, embeddable widgets, and smart digest emails (v1.4–v2.0) are a growth-team fantasy for a network with 10 contacts. Building viral loops before there is anyone to be viral with is the canonical ship-fast anti-pattern. None of it should appear before the network has real RSVPs and real chapters flowing through the DB.
**Sequencing:** HARD CONSTRAINT — the breach lockdown ships FIRST, before any feature, and before any new table. Reasons: (1) the contacts PII+token leak is live and reproduced; (2) every new public-insert feature (rsvps, chapters, event_speakers) inherits the contacts RLS pattern, so the pattern must be CORRECT before it is copied four times. Locking contacts first means the new tables are born INSERT-only-correct instead of cloning the leaky template. This is the one place I concede to Risk/Platform over my original 'v1.0 is already shippable' stance.

Capture-schema-baseline (0001_baseline.sql) must land in the SAME v1.0 as the lockdown, because you cannot safely change RLS policies you haven't first recorded.

rls-probe.sh must exist BEFORE the lockdown change so you can prove the fix (red→green) and BEFORE any new table so each new insert-only table gets a probe assertion. It is the only CI check that must gate v1.0; everything heavier (Playwright/Lighthouse/axe) waits for v2.0.

RSVP (v1.1) before structured content (v1.2) before chapters (v1.3): RSVP is the highest user value and the simplest new table, so it validates the locked insert-only pattern on the lowest-risk feature first. Structured content reuses that proven pattern; chapters reuse it again and add the chapter_slug column that v1.4 moderation depends on.

events.chapter_slug (added in v1.3) must exist before v1.4's per-chapter approval can mean anything — but full captain RBAC stays deferred; v1.4 only needs the status column + admin allowlist.

The static fallback (EVENT_EXTRAS in v1.2, hardcoded chapter cards in v1.3) must NOT be deleted until v2.0, and only after the DB paths are observed working in prod. Removing fallbacks early is the big-bang-migration risk every other PM's roadmap quietly carries; sequencing the deletes last is the ship-fast safety valve.

Reject any insertion of a Node build step (Growth's build-sitemap.js / per-event static pre-render) into the v1.0–v1.3 window: this repo has no build today, and adding one to chase SEO for a 10-contact network reorders cost ahead of value. If crawlability is wanted, a hand-written sitemap.xml/robots.txt slots in anytime as a zero-dependency static file.
