# 1.2.0 — Decompose the Event + the Moderation Gate

**Effort:** XL  ·  **Slug:** `structured-content-and-moderation`  ·  **Branch:** `release/1.2.0-structured-content-and-moderation`

## Theme

Kill the notes-stuffing and the static EVENT_EXTRAS registry by giving speakers/schedule/sponsors real tables — and ship the events.status moderation gate in the SAME release so anon can never attach child rows to arbitrary events or have hidden rows crawled.

## Council notes

This release resolves the single most-litigated sequencing point in the council. Platform PM and the Risk-critique both objected that MVP v1.2 and Risk v1.2 ship open insert-only child tables keyed off events.id while events has NO status column — a defacement/spam vector where anon attaches speaker/sponsor rows to any event. Their fix (status enum + public-reads-approved BEFORE OR WITH the child tables) is adopted: status and the child tables ship together. The Risk-critique's point that HIDDEN_EVENT_IDS at events/index.html:3414 is 'pure security theater' (hidden rows still readable over REST) is honored by replacing it with server-side RLS filtering, not a client list. I adopted MVP's DB-first-with-static-fallback pattern as the non-negotiable risk-reducer (no big-bang cutover; id 173 keeps working) and explicitly DEFERRED deleting js/event-extras.js to v2.0 per the MVP-critique's 'do not remove fallbacks until proven in prod' rule. On the Edge Function: Platform PM wanted ALL writes routed through it from v1.0; the MVP/Growth critiques wanted no gateway. I split the difference — the gateway is introduced HERE, exactly when the first new public-write tables appear and validation/moderation genuinely matter, satisfying Risk/Platform's 'never a raw anon INSERT on new tables' rule without front-loading it onto v1.0. Growth's per-event OG/JSON-LD share targets are unblocked here (they need structured data + approved-status to exist, per the Risk-critique that crawling must wait for the status column), landing the dynamic-event SEO that v1.0 deliberately deferred. The /admin allowlist is an acknowledged stopgap (Platform-critique calls it config-in-code debt) — superseded by real RBAC in v1.4.
