## Council notes

Genuine RBAC was the most expensive thing anyone proposed and the MVP-critique argued hardest to defer it — correctly noting you don't need memberships to capture RSVPs or list chapters. So I sequenced it LATE (v1.4, after RSVP/identity already shipped on the cheaper allowlist), honoring MVP's 'defer RBAC to the end' while still landing it before v2.0. But I adopted Community PM's central thesis over a pure-security framing: RBAC policies with no console to exercise them are dead code, so memberships + the Captain's Console ship in the SAME release (Community v1.2). I resolved the moderation-architecture fight — MVP/Risk's single global /admin vs Community's per-chapter captain review — exactly as the Community-critique demanded: per-chapter captain review becomes the PRIMARY path, with the v1.2 global /admin demoted to fallback for unclaimed 'Other Events', so chapters can actually run themselves. Platform PM's SECURITY DEFINER role function (not a hardcoded UID list) replaces the v1.2 stopgap, satisfying the Platform-critique that hardcoded admin UIDs defeat the 'non-engineer can operate it' goal everyone shares. Follow/membership stays email-first per the Growth-critique (auth-gating follow empties the feed), but routes through the Edge Function so it is not the leaky pattern Risk/Platform objected to. Captain health stats default private per the Community-critique's warning against a public leaderboard demoralizing small chapters.
