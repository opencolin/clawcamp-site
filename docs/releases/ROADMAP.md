# ClawCamp Canonical Roadmap — v1.0 → v2.0

> Decided by the PM Council (5 modes: MVP, Growth, Platform, Community, Risk →
> cross-critique → Chief-of-Staff synthesis). Council workflow run: `wf_3977ba4a-338`.
> Full deliberation in `docs/orchestration/pm-council.md`.

## Converged thesis

The five proposals collapse onto one critical path that every PM either demanded or conceded: lock the live PII/token breach FIRST (it is reproduced — anon can SELECT contacts incl. verification_token via return=representation at js/supabase.js:30/44, and PATCH any contact by email at dashboard/index.html:810/868), establish versioning hygiene in the same release, then build the relational spine (chapters -> structured event content -> identity) BEFORE the features that depend on it, gate public writes through a moderation status column + admin review BEFORE any public child-table or RSVP insert can be trusted, and only retire the static fallbacks (js/event-extras.js, hardcoded chapter cards) in the final release after the DB paths are proven in prod. v2.0 is the milestone where ClawCamp is a real, self-serve, self-moderating multi-chapter network with first-class chapters/memberships/RSVPs and a non-engineer operator runbook. Disagreements resolved as follows: (1) RLS lockdown beats MVP's 'already shippable' framing — it is v1.0 and non-negotiable, but kept afternoon-scale (drop anon SELECT/UPDATE on contacts, switch to return=minimal, auth.uid()-bound dashboard update) WITHOUT building Platform's full Edge-Function gateway up front. (2) Crawlability (Growth) is real and independent of policy work, so robots.txt + sitemap.xml + per-page OG/canonical ship alongside lockdown in v1.0 — but the Node build-step / per-event static pre-render is deferred; sitemap is a tiny commit-time script, and dynamic-event share targets wait until v1.2 when structured data exists. (3) events.status + RLS-filtered public reads (replacing the client-side HIDDEN_EVENT_IDS leak at events/index.html:3414) ship in v1.2 WITH the first structured child tables, so anon can never attach speaker/sponsor rows to arbitrary events or have hidden rows crawled. (4) RSVP and follow are anonymous-email-first (Growth/MVP/Community win over Platform's auth-gate) to protect conversion, but routed through a validating Edge Function with honeypot so they are not raw clones of the leaky pattern (Risk/Platform win on the mechanism). (5) Real memberships/RBAC + the Captain's Console ship together in v1.4 (Community's 'policies need a UI' point) replacing the v1.2 admin-allowlist stopgap. (6) a11y/perf/CSP/CI are folded in as cumulative exit criteria from v1.1 onward (rls-probe + smoke gate from v1.1; Lighthouse/axe/CSP hardened by v2.0) rather than consuming a whole release. Growth's referral flywheel/leaderboards/embeddable widget are deferred to a post-v2.0 backlog note, not numbered, since they presume a network that does not yet exist.

### 1.0.0 — Stop the Bleed + Version Zero  `[M]`
**Theme:** Kill the live PII/token breach and account-takeover vector, establish versioning hygiene, and start the SEO clock — all in one afternoon-scale release that touches data policy and pure-static meta but adds no new feature tables.
**Depends on:** nothing (can start immediately)
**Plan doc:** [`docs/releases/1-0-lockdown-and-versioning.md`](./1.0.0-lockdown-and-versioning.md)

Features:
- Capture the CURRENT live schema as supabase/migrations/0001_baseline.sql (dump of events + the 44-column contacts god-table exactly as they exist, confirmed via anon probe) so schema lives in git before any RLS change — verified there is zero SQL in the repo today
- Enable RLS on contacts: revoke anon SELECT entirely; revoke anon UPDATE entirely; keep anon INSERT only (forms must keep working). Authed users read only their own row via auth.jwt() email match
- Stop leaking tokens: change Prefer header in js/supabase.js:30 from return=representation to return=minimal so verification_token/magic_link_token never cross the wire (currently read at js/supabase.js:44); send-verification edge function reads the token server-side (function stays Supabase-side, not in repo)
- Rewrite the two dashboard PATCH-by-email fetches (dashboard/index.html:810 and :868) to an authenticated session-bound update keyed on auth.uid()/jwt-email, so a visitor can no longer overwrite another person's name/bio/username/prefs
- Add VERSION file (1.0.0) and CHANGELOG.md seeded from git log of existing commits; adopt the /ship flow (hand-bump on every release) — flat-file, no semantic-release tooling yet
- Add js/config.js consolidating the duplicated SUPABASE_URL/SUPABASE_ANON_KEY (currently copy-pasted in js/supabase.js:3-4, js/auth.js:3-4, events/index.html, events/detail/index.html); include it before the other scripts
- Add scripts/rls-probe.sh (curl-based) asserting anon CANNOT read contacts and CANNOT PATCH by email — proves the fix red->green and becomes the reusable guard for every future insert-only table
- Add scripts/smoke.sh (6-line curl+grep of /, /events, /chapters, /events/detail/?id=173 for HTTP 200 + a known string)
- Crawlable foundation (independent of policy, ships in parallel): hand-written robots.txt + a commit-time scripts/build-sitemap.js (zero new runtime build step) emitting sitemap.xml for /events, /chapters, /curriculum, /speakers, /sponsors and the hand-built static event pages; add og:image/twitter:image/canonical meta blocks (currently 0 in index.html) to index, /events, /chapters, /curriculum, /speakers, /sponsors and the 7 hand-built event pages, using one branded 1200x630 default card in /images
- Move the stray scratch surfaces out of the deploy root into /docs/archive: ClawCamp-RSVP-PRD.md, FINAL-PRD.md, HackMeBaby-PRD.md, logo-concepts.html, and the /safemode + /zusemite directories (cleanUrls makes these all live, crawlable URLs today)
- Write SECURITY.md (anon key is public by design; every policy assumes hostile clients) and /docs/data-model.md enumerating the contacts god-table, the title='username:' hack (dashboard/index.html:775/808), and the events.notes text-stuffing as documented debt with owners

_Council notes:_ This is the one release every PM converged on. Risk PM and Platform PM both demanded lockdown-first and I merged their identical 'Stop the Bleed' / 'Ground Truth' work into a single co-owned v1.0 (they conceded the schema baseline must land WITH the lockdown so policies are recorded before they change). MVP PM's 'declare it shippable' framing was overruled by the reproduced breach — but I honored MVP/MVP-critique by keeping the fix afternoon-scale: drop anon SELECT/UPDATE + return=minimal + auth.uid()-bound dashboard update, NOT Platform's full submit-form Edge-Function gateway (deferred until spam actually appears, per MVP's and the Growth-critique's objection that the gateway is heavier than the breach requires and the function source is not even in this repo). Growth PM's crawlability is the one feature folded into v1.0 because it touches no policy and starts the SEO clock day one — but Growth's Node build-pipeline / per-event static pre-render was cut to a tiny commit-time sitemap script per the MVP-critique's 'this repo has no build today' objection; dynamic-event share targets wait for v1.2. rls-probe.sh is intentionally written BEFORE the lockdown change so the fix is provable, and becomes the template every later insert-only table must clone, resolving the Risk-critique that 'mirror the existing pattern' would otherwise propagate the broken policy.

- Add .github/workflows/ci.yml that runs on every PR and on push to master BEFORE Vercel promotes: scripts/rls-probe.sh (REQUIRED check — build fails if anon can read contacts or PATCH by email) + scripts/smoke.sh; turn off blind auto-promote of master via Vercel deploy protection requiring the check

_Council notes:_ Platform PM, Community PM, MVP PM and Growth PM all proposed a chapters table with near-identical columns; I converged them onto ONE migration. The sharpest disagreement — MVP/Growth's events.chapter_slug string vs Platform/Community's chapter_id integer FK — is resolved in favor of the FK, because both Community and Platform critiqued the slug as reintroducing the exact 'indexOf(san francisco) || indexOf(sf)' city-substring drift the app does today; slug stays only as the URL key. Chapters land in v1.1 (not later) because Platform's dependency chain is correct: everything downstream (per-chapter pages, memberships, follows, RSVP auto-follow, sitemap per-chapter URLs) FKs to a chapter, so it is the spine and must precede them. I honored the Growth-critique's CRITICAL CONFLICT — v1.0's lockdown breaks the existing anon-contacts follower count at chapters/index.html:344 — by making the real public-readable aggregate count ship in the SAME release that removes the dead read, so social proof never goes dark. The Risk-critique's demand that the rls-probe become a required CI gate by v1.1 (not v2.0) is honored here, but I sided with the MVP-critique against Risk's full Playwright/Lighthouse/axe wall this early: only the curl-based rls-probe + smoke gate ships now; the heavier suite is staged into later exit criteria.

### 1.2.0 — Decompose the Event + the Moderation Gate  `[XL]`
**Theme:** Kill the notes-stuffing and the static EVENT_EXTRAS registry by giving speakers/schedule/sponsors real tables — and ship the events.status moderation gate in the SAME release so anon can never attach child rows to arbitrary events or have hidden rows crawled.
**Depends on:** 1.1.0
**Plan doc:** [`docs/releases/1-2.0-structured-content-and-moderation.md`](./1.2.0-structured-content-and-moderation.md)

Features:
- Add events.status enum (draft|submitted|approved|rejected, default submitted) replacing the 'STATUS:' string stuffed into notes (submit-event/index.html:817); public /events and /events/detail reads filter to status=approved via RLS — this REPLACES and DELETES the client-side HIDDEN_EVENT_IDS hide-list (events/index.html:3414) which leaks every hidden row over REST today
- Create event_speakers (event_id FK, name, role, org, photo_url, bio, sort_order), event_schedule/schedule_blocks (event_id FK, start_time, end_time, title, sort_order), and event_sponsors (event_id FK, sponsor_name, sponsor_url, tier) — all public SELECT, writes routed through a single validating submit-form Edge Function (length caps, email/URL format, reject HTML, server-side normalizeUrl, honeypot) so they are NOT raw clones of the leaky anon-insert pattern
- Rewrite submit-event/index.html saveEvent (lines ~788-821) to INSERT child rows into these tables via the Edge Function instead of concatenating into events.notes; reserve notes for free-text reviewer comments only
- Rewrite events/detail/index.html to read speakers/schedule/sponsors from the new tables by event_id FIRST, falling back to window.EVENT_EXTRAS (js/event-extras.js:117 include) ONLY when no DB rows exist — so the hand-built id 173 entry keeps working with ZERO migration
- One-time backfill: migrate the single js/event-extras.js id-173 entry into the new tables (do NOT delete the registry yet)
- Build ONE thin /admin review page (auth + admin-allowlist RLS, a small set of admin user IDs as an interim stopgap) listing submitted events and flipping status to approved/rejected; rejection reason writes to notes
- Now that structured data and approved-status exist, extend the sitemap and add full per-event OG/JSON-LD share targets for approved dynamic events (deferred from v1.0): generate a static /events/<slug>/ meta page per approved event in the commit-time build script, or a Vercel OG endpoint — only approved rows, so drafts/spam are never crawled
- Add a Playwright stored-XSS regression test: submit an event named <img src=x onerror=alert(1)> and assert it renders as inert text on /events and /events/detail (locks in the textContent safety)

_Council notes:_ This release resolves the single most-litigated sequencing point in the council. Platform PM and the Risk-critique both objected that MVP v1.2 and Risk v1.2 ship open insert-only child tables keyed off events.id while events has NO status column — a defacement/spam vector where anon attaches speaker/sponsor rows to any event. Their fix (status enum + public-reads-approved BEFORE OR WITH the child tables) is adopted: status and the child tables ship together. The Risk-critique's point that HIDDEN_EVENT_IDS at events/index.html:3414 is 'pure security theater' (hidden rows still readable over REST) is honored by replacing it with server-side RLS filtering, not a client list. I adopted MVP's DB-first-with-static-fallback pattern as the non-negotiable risk-reducer (no big-bang cutover; id 173 keeps working) and explicitly DEFERRED deleting js/event-extras.js to v2.0 per the MVP-critique's 'do not remove fallbacks until proven in prod' rule. On the Edge Function: Platform PM wanted ALL writes routed through it from v1.0; the MVP/Growth critiques wanted no gateway. I split the difference — the gateway is introduced HERE, exactly when the first new public-write tables appear and validation/moderation genuinely matter, satisfying Risk/Platform's 'never a raw anon INSERT on new tables' rule without front-loading it onto v1.0. Growth's per-event OG/JSON-LD share targets are unblocked here (they need structured data + approved-status to exist, per the Risk-critique that crawling must wait for the status column), landing the dynamic-event SEO that v1.0 deliberately deferred. The /admin allowlist is an acknowledged stopgap (Platform-critique calls it config-in-code debt) — superseded by real RBAC in v1.4.

### 1.3.0 — On-Site RSVP, Real Profiles, and Storage  `[XL]`
**Theme:** Capture the highest-value action on-site instead of bouncing to Luma, give users a first-class identity row instead of the contacts-title hack, and stand up one scoped Storage bucket shared by avatars and (later) recaps.
**Depends on:** 1.2.0
**Plan doc:** [`docs/releases/1-3.0-rsvp-and-identity.md`](./1.3.0-rsvp-and-identity.md)

Features:
- Create rsvps table (id, event_id FK, email, name, status enum going|waitlist|cancelled, created_at) routed through the v1.2 submit-form Edge Function with honeypot — anonymous-email-first (NOT auth-gated) so RSVP conversion is not cratered, with optional account linkage when a session exists
- Add an inline email-capture RSVP form on events/detail/index.html alongside the existing Luma button (currently the only CTAs at :238 and :346), with a 'You're going' return state; Luma stays as the secondary 'add to calendar / external ticketing' action for is_external events
- Show a live attendee count on the detail page via a count=exact head request on rsvps for that event_id ('23 going'), plus an 'Add to Calendar' (Google/Apple/.ics) control next to RSVP
- Create profiles table keyed to auth.users.id (id uuid, username citext UNIQUE, display_name, bio, links jsonb, photo_path, created_at; owner-write, public-SELECT RLS); migrate signed-in users' name/bio/links and the title='username:' values out of contacts
- Replace the LIKE-on-contacts.title username check (dashboard/index.html:775) with a real UNIQUE constraint on profiles.username and a clean availability query; rewrite the dashboard save (dashboard/index.html:808) to write profiles, not the title hack; contacts becomes strictly CRM/lead-capture
- Provision ONE Supabase Storage bucket 'media' with per-user-folder RLS + server-side content-type/size validation; wire dashboard's currently preview-only photo upload (profile-photo block at dashboard/index.html:587) to actually upload and persist photo_path
- Post-RSVP 'invite a friend' prompt with a prefilled share link carrying UTM/ref params; add UTM capture in the Edge Function path (read window.location.search, write utm_source/utm_medium/utm_campaign/ref onto the row) so RSVPs and signups are attributable

_Council notes:_ Two camps clashed hardest here and I split the decision on the merits of each half. On auth-gating: Platform PM and Risk PM wanted rsvps.profile_id FK'd to auth.users with login required; Growth PM, MVP PM and Community PM warned this craters conversion versus the frictionless Luma click. I sided with the growth camp on the FRONT DOOR — RSVP and follow accept anonymous email — because forcing magic-link before 'I'm going' is how you lose a 10-contact network. But I sided with Platform/Risk on the MECHANISM: the row routes through the v1.2 validating Edge Function with a honeypot, so it is not the raw leaky anon-insert pattern they rightly objected to, and optional account linkage preserves the upgrade path. Platform PM's strongest unique contribution — a real profiles table replacing the title='username:' hack (dashboard/index.html:775/808) with a UNIQUE constraint — is adopted in full; Community PM had no first-class identity row and the Platform-critique correctly flagged that RBAC/RSVP/follows need a trustworthy owner to FK to. Storage is provisioned ONCE here (Community-critique's explicit cross-mode coordination ask) so avatars now and recaps in v1.5 share one bucket with per-user/per-chapter RLS, rather than three roadmaps each standing up their own. Growth's UTM capture and post-RSVP invite prompt land here because the Growth-critique is right that they must arrive WITH the first share surface or early shares are unattributable — but the referral leaderboard/K-factor machinery stays deferred (no network to be viral with yet).

### 1.4.0 — Real Roles + the Captain's Console  `[XL]`
**Theme:** Turn the cosmetic 'captain' badge into enforced, scoped permissions and ship the console that exercises them — the first time captains can run their own chapter, replacing the v1.2 admin-allowlist stopgap.
**Depends on:** 1.3.0
**Plan doc:** [`docs/releases/1-4.0-rbac-and-captains-console.md`](./1.4.0-rbac-and-captains-console.md)

Features:
- Create memberships table (profile_id FK, chapter_id FK, role enum member|captain|admin, created_at) exposed via a SECURITY DEFINER role helper usable in RLS — replacing the v1.2 hardcoded admin-UID allowlist (config-in-code debt) so new admins/captains are data, not deploys
- Wire the existing join modal (chapters/index.html openJoin/submitJoin at :431/:452) and follow action to write a chapter_follows / memberships row instead of the contacts?form_type=chapter_follow hack — email-first via the Edge Function, with optional account linkage (a follow must work without login)
- Make /dashboard 'My Chapters' (dashboard/index.html, currently an empty-state placeholder) list the member's real chapters with an unfollow control and links to each chapter page; drive the dash role badge from memberships (a server fact, not a client guess), and a member cannot self-promote (only the role function grants roles)
- Build the Captain's Console in /dashboard (gated by memberships.role=captain for that chapter_id via RLS): edit their own chapter page content, view/edit/cancel events where events.chapter_id matches their chapter (reusing the 4-tab submit-event form), and approve/reject SUBMITTED events for THEIR chapter — making per-chapter review the PRIMARY path with the v1.2 global /admin as fallback for unclaimed/Other Events
- Surface a captain-facing RSVP roster (headcount vs capacity + CSV export for door check-in) in the console, reading the rsvps table; add a capacity field to events for waitlist logic
- Add a per-chapter follow feed: upcoming approved events from chapters the user follows, surfaced on the dashboard as 'From your chapters'

_Council notes:_ Genuine RBAC was the most expensive thing anyone proposed and the MVP-critique argued hardest to defer it — correctly noting you don't need memberships to capture RSVPs or list chapters. So I sequenced it LATE (v1.4, after RSVP/identity already shipped on the cheaper allowlist), honoring MVP's 'defer RBAC to the end' while still landing it before v2.0. But I adopted Community PM's central thesis over a pure-security framing: RBAC policies with no console to exercise them are dead code, so memberships + the Captain's Console ship in the SAME release (Community v1.2). I resolved the moderation-architecture fight — MVP/Risk's single global /admin vs Community's per-chapter captain review — exactly as the Community-critique demanded: per-chapter captain review becomes the PRIMARY path, with the v1.2 global /admin demoted to fallback for unclaimed 'Other Events', so chapters can actually run themselves. Platform PM's SECURITY DEFINER role function (not a hardcoded UID list) replaces the v1.2 stopgap, satisfying the Platform-critique that hardcoded admin UIDs defeat the 'non-engineer can operate it' goal everyone shares. Follow/membership stays email-first per the Growth-critique (auth-gating follow empties the feed), but routes through the Edge Function so it is not the leaky pattern Risk/Platform objected to. Captain health stats default private per the Community-critique's warning against a public leaderboard demoralizing small chapters.

### 2.0.0 — The Self-Running Multi-Chapter Network  `[XL]`
**Theme:** ClawCamp is a real, self-serve, self-moderating multi-chapter platform: static data fully retired, self-serve chapter provisioning, map+search discovery, and the full quality wall (a11y, perf, CSP, monitoring) enforced so a non-engineer can run it end to end.
**Depends on:** 1.5.0
**Plan doc:** [`docs/releases/2-0-self-running-platform.md`](./2.0.0-self-running-platform.md)

Features:
- Self-serve chapter provisioning: the existing 'Start a Chapter' application (chapters/index.html submitChapterApp at :382, form_type=chapter_application) on approval AUTO-CREATES a chapters row and assigns the applicant the captain membership role — no manual DB editing
- Retire all static data NOW that DB paths are proven in prod: delete js/event-extras.js and its include (events/detail/index.html:117), remove the hardcoded chapter-card fallbacks, and confirm zero events.notes content-stuffing remains; any remaining hand-built static event pages either read from tables or are documented as grandfathered
- Map + search discovery on /events and /chapters reusing js/clawcamp-map.js from the homepage hero (closing the 'no map on /events, no search across events' gaps), plotting chapters/events by lat/lng
- Engagement notifications driven off memberships + rsvps and the existing dashboard email prefs (sub_events/sub_newsletter): new-event-from-a-followed-chapter and RSVP reminders, with a notification-preferences center protecting deliverability
- Opt-in per-chapter member directory on chapter pages using stored profile photos/@username/bio (turning chapters into networks), and a private captain health dashboard (follower growth, RSVP conversion, attendance trend) — public momentum framed as 'this city is forming', never a competitive ranking
- Full quality wall enforced in CI: Lighthouse budgets (Perf>=90, A11y>=95, Best-Practices>=95) on /, /events, /chapters, /events/detail; axe-core in Playwright for the join modal + start-a-chapter form (keyboard, focus rings, aria-labels, alt text on currently alt='' images); a Content-Security-Policy header in vercel.json authored AFTER the share/calendar/map surfaces exist so it allowlists them; split the 198KB index.html static event list into a Supabase-fed lazy render
- Production-grade observability: client-side error reporting (Sentry) in js/*, Supabase log alerting on anomalous read/insert spikes, Dependabot/SRI re-pinning on the pinned @supabase CDN dep, a backup+restore drill with documented RTO/RPO, a /privacy page + data-retention policy for contacts, and a final full anon re-probe confirming NO PII is reachable anywhere
- RLS coverage audit as a checked-in test asserting EVERY table (events, chapters, event_speakers, schedule_blocks, event_sponsors, profiles, memberships, rsvps, chapter_follows, contacts) has explicit SELECT/INSERT/UPDATE/DELETE policies and no anon write leaks; checked-in supabase gen types + ERD; indexes on hot FKs (events.chapter_id, event_speakers.event_id, schedule_blocks.event_id, rsvps.event_id, chapter_follows.chapter_id) with a query budget
- Write a one-page operator runbook in /docs (submit event -> approve in console/admin -> add a chapter -> read RSVPs -> compose a recap) and cut a tagged v2.0.0 release with /document-release and a CHANGELOG complete back to v1.0

_Council notes:_ v2.0 is the milestone the prompt requires: ClawCamp is a real multi-chapter platform, not hardcoded. The static-data retirement is sequenced LAST per the MVP-critique's explicit safety valve — fallbacks (EVENT_EXTRAS, hardcoded cards) are deleted only after the DB paths are observed working in prod, avoiding the big-bang migration risk other roadmaps quietly carried. Self-serve provisioning (Community v2.0) is correctly last because it depends on the entire stack: chapters schema (v1.1), memberships role (v1.4), and the console (v1.4). The Risk PM's a11y/perf/CSP wall is folded in here as EXIT CRITERIA rather than a deferred release, honoring the prompt's instruction — but I adopted both the Growth-critique's and Platform-critique's warning that CSP must be authored AFTER the share/calendar/map/embed surfaces exist so it allowlists them instead of silently killing virality, which is precisely why it lands in the final release. The full observability/backup/privacy baseline (Risk v2.0) and the RLS coverage audit + indexes (Platform v2.0) consolidate here as the provable 'hardened baseline'. Growth's referral flywheel, public leaderboards, K-factor dashboards, and the embeddable cross-origin widget were the most-objected-to items in the council (MVP, Community, and Risk all flagged them as premature, demoralizing, or an un-gated XSS/exfil vector) — they are deliberately NOT numbered into this roadmap and belong in a post-2.0 growth backlog, to be revisited only once there is a real network with real RSVPs and chapters flowing through the DB.
